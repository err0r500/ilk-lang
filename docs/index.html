ilk

Define your types. Name your entities. Trace your data.

ilk is a single-file data modeling language. One .ilk file holds both type declarations — the abstract vocabulary of a domain — and instance bindings — the named entities that actually exist in your system.

Beyond shape validation, ilk tracks data provenance: each field must be traceable back to a declared source. The compiler catches missing mappings before runtime.

.ilk — types & instances validates shape + data flow

1 The basics — types and instances

Type declarations describe shapes (what fields exist, what types they have). Instance bindings name the concrete entities in your domain — not runtime values, but the fixed catalog of things your system knows about. Both live in one file.

domain.ilk
// ── Type declarations ──────────────────────────
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE"

type User = {
    id    Uuid
    name  String
    email String
}

// ── Instance bindings ───────────────────────────
// Named entities in this specific domain

adminUser = User {
    id    Uuid
    name  String
    email String
}

guestUser = User {
    id    Uuid
    name  String
    email String
}
Instance bindings are not runtime values. adminUser doesn't hold a real UUID or name — it declares that your domain has an entity called adminUser whose id field will be a Uuid at runtime. Think of it as a typed slot catalog.

2 How tight is a value?

ilk has three constraint levels for field values. They form a tightening progression: from fully open to exact match.

In a typeIn an instanceMeaning
String String Open — any string value at runtime
Concrete<String> "webhook" Instance-fixed — the instance author picks one specific value
"POST" "POST" Type-fixed — only this exact value is ever valid
type
type Webhook = {
    // any URL at runtime
    url     String

    // instance author fixes the event name
    event   Concrete<String>

    // always exactly "POST"
    method  "POST"
}
✓ valid instance
userCreatedHook = Webhook {
    // open: leave as String (value comes at runtime)
    url     String

    // instance-fixed: pick the event name
    event   "user.created"

    // type-fixed: must be exactly "POST"
    method  "POST"
}

3 Shapes — structs, lists, unions

Structs describe field collections. Lists carry cardinality. Unions let a field accept multiple alternatives.

Structs
// anonymous: 1 field of any name/type
{_}

// anonymous with type constraint
{_ String}

// open: any fields
{...}

// named fields
{id Uuid, name String}

// any fields + required id
{...} & {id Uuid}
Unions
// literal union
type HttpMethod = "GET" | "POST" | "PUT"

// identifier union
type Status = Pending | Active | Archived

// mixed
type Tag = {_ String} | Concrete<String>
Lists
// 0 or more
[]Event

// exactly 3
[3]Tag

// 1 or more
[1..]Tag

// 2 to 5
[2..5]Tag

// up to 10
[..10]Tag

// ── References ─────────────────
// point to a binding, no data flow
&Event
[]&Event

// refinable reference
// (can override open fields with literals)
-Event
[]-Event

4 Data flow — where does this field come from?

This is the unique part. @source declares which upstream fields can supply data to a downstream field. Every non-Concrete field in the downstream struct must be explicitly traced back — the compiler enforces it.

type with @source
type DbLookup = {
    args    {...}

    // @out: this field provides data to callers
    @out
    returns {...}
}

type Endpoint = {
    params  {...}

    // db can only use data from params
    @source [params]
    db      DbLookup

    // response can use params OR db.returns
    @source [params, db.returns]
    response {...}
}
✓ all fields traced
getUser = Endpoint {
    params  {id Uuid}

    db findUserDb & {
        // maps params.id → db arg
        userId Uuid = params.id  ✓
    }

    response {
        id   Uuid   = db.returns.id    ✓
        name String = db.returns.name  ✓
    }
}
✗ untraced field → compile error
getUser = Endpoint {
    params  {id Uuid}
    db      findUserDb & {
        userId Uuid = params.id
    }

    response {
        id   Uuid   = db.returns.id
        // name has no = assignment
        name String  ✗ @source violation
    }
}
Why does this matter? In a large API codebase, it's easy to accidentally include data in a response that shouldn't be there (e.g. internal IDs leaking out). @source makes the data flow explicit and machine-checked — every field in a response must be justified.

5 Annotations

Annotations go on the line immediately before what they annotate.

AnnotationWhereWhat it does
@maininstanceEntry point — the root binding the compiler validates from
@source [fields]field in typeData provenance: this field's values must trace back to fields
@outfield in typeOutput field — exempt from @source checks; can be referenced by callers
@assoc [T]typeInstances carry typed associations (used with @constraint)
@constraint exprfield in typeBoolean predicate evaluated at validation time
@doc "…"fieldImplementation hint, preserved in AST for downstream tooling

6 Full example — REST API with data flow

Type the whole API domain in one file: method enum, DB layer, response shapes, endpoint shape, and the concrete endpoints that exist in your system. The compiler validates structure and traces every response field back to a source.

api.ilk
// ── Types ───────────────────────────────────────────────────────

type HttpMethod = "GET" | "POST" | "PUT" | "DELETE"

type DbMethod = {
    name    Concrete<String>   // instance author picks the method name
    args    {...}

    @out                       // result flows outward to callers
    returns {...}
}

type HttpResponse = {
    status  Concrete<Int>      // instance author picks status code
    body    {...}
}

type Endpoint = {
    @constraint forall(templateVars(path), v => v in keys(params))
    path    Concrete<String>

    method  HttpMethod
    params  {...}
    body    {...}

    @source [params, body]    // db args must come from request
    db      DbMethod

    @source [params, body, db.returns]  // responses may use db results too
    responses []HttpResponse
}

type Api = {
    endpoints []Endpoint
}


// ── Instances ───────────────────────────────────────────────────

findUser = DbMethod {
    name    "users.findById"
    args    {userId Uuid}
    returns {id Uuid, name String}
}

getUser = Endpoint {
    path    "/users/{id}"
    method  "GET"
    params  {id Uuid}
    body    {}

    db findUser & {
        userId Uuid = params.id   // maps request param → db arg
    }

    responses [
        {
            status  200              // Concrete → exempt from @source
            body    {
                id   Uuid   = db.returns.id
                name String = db.returns.name
            }
        }
        {
            status  404
            body    {message "User not found"}
        }
    ]
}

@main
api = Api {
    endpoints [getUser]
}

7 Full example — event sourcing with tag constraints

@assoc attaches typed metadata to instances. @constraint validates relationships at compile time — here, every event in a query must carry every tag the query declares.

types
// Tag: parametrized {field String} or a unique string
type Tag = {_ String} | Concrete<String>

// Event carries open fields + a timestamp
// Instances can declare Tag associations
@assoc [Tag]
type Event = {...} & {timestamp Int}

// Every event must carry every listed tag
type QueryItem = {
    @constraint forall(tags, t =>
        forall(eventTypes, e => e.assoc(t)))
    eventTypes []&Event
    tags       []Tag
}

type Command = {
    fields {...}
    @source [fields]
    emits  []Event
    query  []QueryItem
}

@main
type Board = {commands []Command}
instances
// Define reusable tags
userIdTag   = Tag {userId String}
userNameTag = Tag {name   String}

// Event associated with both tags
userRegistered = Event<userIdTag, userNameTag> {
    id        String
    name      String
    timestamp Int
}

// registerUser command: fields flow into emitted event
registerUser = Command {
    fields {userId String, name String}
    emits  [
        userRegistered & {
            id   String = fields.userId
            name String = fields.name
        }
    ]
    query  [
        {
            eventTypes [&userRegistered]
            tags       [userIdTag]
        }
    ]
}

@main
board = Board {
    commands [registerUser]
}
